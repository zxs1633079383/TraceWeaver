## 2. FeedbackLog 设计

### 数据结构

```typescript
interface FeedbackEntry {
  id: string           // UUID
  seq: number          // 自增序号，跨重启连续
  ts: string           // ISO 8601
  harness_id: string
  entity_id: string
  entity_type: string
  trigger_state: string
  result: 'pass' | 'fail' | 'skipped'
  reason: string       // refs_checked[0].note 或 LLM 返回摘要
  duration_ms: number
}
```

### 核心 API

```typescript
class FeedbackLog {
  load(): void                                        // 重放 NDJSON
  record(input: FeedbackRecordInput): FeedbackEntry  // append
  query(params: FeedbackQuery): FeedbackEntry[]       // 过滤查询
  getSummary(harness_id: string): HarnessFeedbackSummary
  getAllSummaries(): HarnessFeedbackSummary[]
}

interface HarnessFeedbackSummary {
  harness_id: string
  total: number
  pass: number
  fail: number
  skipped: number
  failure_rate: number              // fail/total
  consecutive_failures: number      // 尾部连续失败次数
  recent_reasons: string[]          // 最近 3 次失败原因
  trend: 'improving'|'degrading'|'stable'|'unknown'
  last_evaluated: string
}
```

### 趋势算法

将历史均分为两半，比较前半与后半的失败率：
- `recentFail > prevFail + 0.1` → `degrading`
- `recentFail < prevFail - 0.1` → `improving`
- 否则 → `stable`（少于 4 条数据 → `unknown`）

## 3. HarnessValidator 设计

### 检测项

| 类型 | 触发条件 | severity |
|------|---------|----------|
| `orphaned_ref` | entity.constraint_refs 中 id 在 HarnessLoader 中不存在 | error |
| `dead_harness` | harness.applies_to 的实体类型在注册表中无实体 | warning |
| `persistent_failure` | feedbackLog.getSummary(id).consecutive_failures ≥ threshold | warning |

### AlignmentIssue 结构

```typescript
interface AlignmentIssue {
  severity: 'error' | 'warning'
  type: 'orphaned_ref' | 'dead_harness' | 'persistent_failure'
  harness_id?: string
  entity_id?: string
  message: string       // 人类可读，中文
  suggestion?: string   // 修正建议
}
```

### 触发时机（自动）

1. Daemon 启动时 → 首次全量校验
2. FsWatcher 检测到 `.traceweaver/harness/` 目录下文件变更 → 重新 scan + validate
3. TriggerExecutor 评估后检测到 persistent_failure → 立即触发单条 validate
4. IPC `harness_validate` 请求 → 按需触发

### 结果输出

- 写入 InboxAdapter（message 含 issue 摘要）
- 发布 `harness.misaligned` 事件到 EventBus（供外部监听）

## 4. OTel 导出适配器设计

### 适配器接口

```typescript
interface TraceExporter {
  readonly name: string
  export(spans: SpanMeta[]): Promise<void>
  shutdown?(): Promise<void>
}
```

### 适配器注册表

```typescript
class ExporterRegistry {
  register(exporter: TraceExporter): void
  async exportAll(spans: SpanMeta[]): Promise<void>  // 并行调用所有已注册适配器
  async shutdown(): Promise<void>
}
```

### 适配器清单

| 名称 | 实现 | 协议 | 端口 |
|------|------|------|------|
| `console` | ConsoleExporter（内置） | - | - |
| `otlp-http` | OtlpExporter（已有） | HTTP/JSON | 4318 |
| `otlp-grpc` | OtlpGrpcExporter（新增） | gRPC/Protobuf | 4317 |

### 环境变量配置

| 变量 | 说明 | 示例 |
|------|------|------|
| `TW_OTEL_EXPORTER` | 逗号分隔的适配器列表 | `otlp-grpc` 或 `otlp-http,console` |
| `TW_OTEL_ENDPOINT` | OTLP 端点（http/grpc 适配器共用） | `http://jaeger:4317` |
| `TW_OTEL_HEADERS` | 附加请求头（JSON 字符串） | `{"x-token":"abc"}` |

### gRPC 适配器实现策略

使用 `@grpc/grpc-js` + `@grpc/proto-loader` 加载 OpenTelemetry OTLP proto 定义：

```
packages/tw-daemon/src/otel/proto/
└── opentelemetry/proto/collector/trace/v1/trace_service.proto
    └── (+ 依赖的 common/resource/trace proto)
```

gRPC Channel 配置：
- `keepalive_time_ms: 30000`
- `keepalive_timeout_ms: 5000`
- TLS：默认关闭（k8s 集群内通信），可通过 `TW_OTEL_TLS=1` 开启

## 7. 非功能需求

- **零停机变更**：所有新组件为可选注入，不影响现有功能
- **无阻塞**：FeedbackLog.record() 同步写磁盘，但不在评估关键路径上阻塞
- **幂等**：HarnessValidator.validate() 可重复调用，无副作用（只读 + 写 inbox）
- **gRPC 错误隔离**：导出失败不影响 Daemon 主功能，仅记录 console.error
